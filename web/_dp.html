    Aspida API & encryption

    How a client talks to the inference server, and how every byte on the wire is sealed. Everything below — transport, handshake, AEAD, the LLM engine — is implemented from scratch in Ada/SPARK with no third-party crypto or ML libraries.

    1 · What Aspida gives you

    Aspida is end-to-end encrypted inference — built on its own engine and its own cryptography, with no third-party crypto libraries and no external services. The model, the cryptography and the transport are all ours, written from the ground up in Ada/SPARK.

    Encrypted end to end
    • Your prompt is sealed inside your browser before it ever leaves the device.
    • The network, the relay and the hosting provider only ever see ciphertext.
    • Forward secrecy — captured traffic stays unreadable into the future.
    • The server is authenticated against a pinned key — no man-in-the-middle.
    • Conversation history is encrypted at rest.
    • Streamed token-by-token, with many sessions served at once by our own batching engine.

    2 · Transport & endpoints

    A browser cannot open raw TCP, so a thin bridge relays a WebSocket to the server's TCP socket — seeing only ciphertext. A native client may skip the bridge and speak TCP directly.

    EndpointPurpose
    GET /serverkeyThe server's pinned static public key (hex). Pin this in the client.
    GET /, /app.js, …Static web client (this demo).
    WS /wsWebSocket ⇄ TCP relay to the server (the encrypted channel rides inside it).
    TCP :8765Direct secure-channel socket for native clients (no bridge).

    Every message — handshake frame or sealed record — is a length-prefixed frame:

    [ 4 bytes, big-endian length N ][ N bytes payload ]

    3 · Handshake

    A single round trip establishes the channel. The client and server exchange ephemeral public keys and derive a fresh set of per-session keys:

    • Server authentication — the server is bound to its pinned public key; the client rejects any other, so there is no man-in-the-middle.
    • Forward secrecy — ephemeral keys on both sides mean recorded traffic stays secret even if a long-term key later leaks.
    • Channel binding — both ends independently derive an identical binding value; a mismatch aborts the session.

    The exact key-schedule construction is withheld while the technology is in development.

    4 · Record encryption

    After the handshake, every record is sealed with authenticated encryption (AEAD). Each direction uses its own key and a unique, non-repeating nonce, and a per-record authentication tag means any tampering makes decryption fail — so both ends reject altered bytes.

    Cipher selection and construction specifics are withheld during development.

    5 · Protocol messages

    Inside each sealed record the first plaintext byte is a tag; the rest is the UTF-8 payload.

    TagByteDirMeaning
    Session's'↔Client: select/resume a session (id, or empty = new). Server: the assigned session id.
    Prompt'p'→The user message (UTF-8). Triggers generation.
    Prefill'.'←Progress tick while the prompt is being ingested.
    Token't'←One generated token piece (UTF-8), streamed live.
    Done'!'←Generation finished.
    Error'e'←Error message (UTF-8).

    6 · A full session

    1.  serverPub ← GET /serverkey            # pin it
2.  open WS /ws  (or TCP :8765)
3.  handshake  (send ePub, recv fPub, derive kC2S/kS2C)   — §3
4.  send  seal('s' + "")                  # new session
    recv  open('s' + "<session-id>")
5.  send  seal('p' + "What is the capital of France?")
6.  recv  open('.')                        # prefill tick(s)
    recv  open('t' + "The")                # tokens, streamed
    recv  open('t' + " capital") …
    recv  open('!')                        # done
    # reuse the same channel for the next turn (send 's' + id earlier to resume history)

    Concurrent sessions are served together by the engine's continuous-batching scheduler: many sessions share each GPU forward pass, so N users cost far less than N× the time. History is stored server-side, encrypted at rest.

    7 · Cryptography

    Built entirely from scratch — no OpenSSL, no libsodium — on standard, well-studied building blocks:

    elliptic-curve key exchange authenticated encryption (AEAD) key derivation encrypted at rest

    Specific algorithm choices and parameters are withheld while the technology is in development.

    Engine: Llama / Qwen / Gemma backends from GGUF, custom GPU inference kernels, continuous batching — all in Ada/SPARK.
